## Purpose

- Why this playbook exists:
  - make test additions and reproduction changes follow the repository's actual harness.
- What change class it standardizes:
  - adding C++ gtests, Python unittests, example smoke checks, or module-specific reproduction coverage.
- What risks it is meant to reduce:
  - putting a test in the wrong binary, creating misleading CTest labels, missing service/package prerequisites, or
    validating only one build system when both CMake and Bazel matter.

## When To Use This Playbook

- Use when:
  - adding or moving tests under `tests`;
  - choosing a validation path for a bugfix;
  - reviewing whether a change has adequate UT, ST, Python, or example coverage.
- Do not use when:
  - the task only changes product code and the module doc already gives exact test commands.
- Escalate to design-first review when:
  - a change needs new test binary partitioning;
  - a new label or prefix convention is proposed;
  - tests require a new service lifecycle, external dependency, or persistent test data model.

## Preconditions

- Required context to read first:

  - relevant `tests/*/CMakeLists.txt` or `BUILD.bazel`;

- Required assumptions to verify before coding:
  - whether the behavior can be isolated in UT or needs ST services;
  - whether Python/example/package validation is required;
  - whether the touched code path is built by both CMake and Bazel.

## Risk Classification

| Risk Area | Question to answer before implementation | Low-risk signal | Escalation signal |
| --- | --- | --- | --- |
| binary placement | Which test binary will own this source file? | nearby tests already compile into the desired target | CMake filters would route the file to a different bucket |
| labels | Which CTest labels will be generated? | existing prefix and directory convention matches intent | new prefix, disabled test, or serial behavior is needed |
| runtime lifecycle | Does the test need worker/master/client processes? | pure local component test | cluster, ETCD, package, Python, or example lifecycle required |
| build systems | Does the area have Bazel and CMake coverage? | touched area is CMake-only or Bazel-only | both build files need aligned targets |
| stability | Can it run under randomized parallel CTest? | no shared ports, global state, or long runtime | needs `EXCLUSIVE_`, generated assets, or high timeout |

## Implementation Plan

1. Choose UT, ST, Python, example, or perf coverage based on the behavior boundary.
2. Put new C++ tests under the directory that maps to the intended binary.
3. Add CMake or Bazel target changes only when existing globs/targets do not already include the file.
4. For ST service ports, use allocator-backed `ExternalClusterTest::GetFreePort()` or the shared shell helper path;
   do not add ad hoc random-port or `netstat` parsing logic.
5. Use `LEVEL1_` or `EXCLUSIVE_` only when runtime, duration, or shared-state behavior requires it.
6. Prefer narrow `ctest -R` validation first, then the relevant label or package-level command.
7. Update module context when the new test reveals a stable reproduction path future agents should reuse.

## Guardrails

- Must preserve:
  - existing prefix semantics for `DISABLED_`, `EXCLUSIVE_`, `LEVEL1_`, and `LEVEL2_`;
  - CTest label behavior generated by `GoogleTestToCTest.cmake`;
  - ST port allocation through the shared allocator and its `/tmp/datasystem-st-ports-${UID}` lease model;
  - Python tests' package and service startup assumptions.
- Must not change without explicit review:
  - generated CTest naming semantics;
  - test binary partitioning in `tests/ut/CMakeLists.txt` or `tests/st/CMakeLists.txt`;
  - global `build.sh -t` behavior.
- Must verify in source before claiming:
  - whether a test is UT or ST;
  - whether `run_python` or `run_example` is part of a normal validation path;
  - whether Bazel test coverage mirrors CMake coverage.

## Validation Plan

- Fast checks:
  - `bash build.sh -t build`
  - `ctest -R <test_suite.test_name>`
- Representative checks:
  - `bash build.sh -t run_cases -l ut`
  - `bash build.sh -t run_cases -l st`
  - `bash build.sh -t run_cases -l "object level0"`
  - `bash build.sh -t run_cases -l "stream level0"`
- Package/runtime checks:
  - `bash build.sh -t run_python`
  - `bash build.sh -t run_example`
- Negative-path verification:
  - check that disabled tests are intentionally disabled;
  - check that long or shared-state tests are `LEVEL1_` or `EXCLUSIVE_` when needed.
